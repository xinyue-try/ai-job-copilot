# AI Job Copilot 作品集 Case Study

## 1. 项目背景

求职准备不是一个单点问题，而是一条连续链路：

- 投递前需要判断 JD 是否匹配。
- 面试前需要准备高风险问题。
- 面试中需要快速组织回答。
- 面试后需要把经验沉淀下来。
- 下一次面试需要复用之前的经验。

很多 AI 工具只能回答“这道题怎么答”，但缺少对个人历史经历的持续记忆。AI Job Copilot 的目标是把求职过程做成一个可持续迭代的闭环。

## 2. 用户痛点

### 痛点一：JD 分析成本高

候选人需要快速判断岗位是否值得投，但 JD 往往信息密度高、要求模糊，手动分析效率低。

### 痛点二：面试准备容易泛泛而谈

直接问大模型容易得到通用回答，缺少和个人简历、项目经历、历史面试表现的结合。

### 痛点三：复盘没有沉淀

面试后的复盘常常散落在聊天记录、备忘录或文档里，下次面试很难被系统地复用。

### 痛点四：AI 容易“替用户编经历”

如果没有明确约束，大模型可能会把用户没有做过的事情写成事实，比如 A/B 实验、增长数据或业务结果。

## 3. 产品方案

AI Job Copilot 用一个闭环解决上述问题：

```text
JD 分析
-> 高风险面试雷达
-> 问一问 AI
-> 面试后复盘
-> Job Memory
-> RAG 召回
-> 下一轮面试准备
```

它的核心不是生成“标准答案”，而是帮助用户把真实经历变成可检索、可复用、可表达的面试材料。

## 4. 核心功能设计

### 4.1 JD 分析

用户粘贴 JD 或上传岗位截图，系统结合简历生成：

- 岗位摘要
- 匹配分
- 主要匹配点
- 风险点
- 高风险面试雷达
- 缺失能力
- 简历优化建议
- 打招呼语

设计重点：不仅告诉用户“匹配不匹配”，还要指出“面试官可能追问什么”。

### 4.2 高风险面试雷达

系统从 JD 和简历差距中生成高风险问题。用户点击问题后，直接跳到“问一问 AI”，并自动填入该问题。

这个交互把静态报告变成了可继续追问的 Copilot。

### 4.3 问一问 AI

用户输入面试问题，系统结合当前 JD、轮次、回答风格、回答时长和历史 Job Memory 生成：

- 30 秒短答
- 完整答法
- 可引用证据
- 不要这样说
- 可能追问
- 本次参考记忆

设计重点：回答需要有结构，但不能编造用户没有的经历。

### 4.4 Job Memory

用户面试后粘贴复盘、ASR 文本或问题清单，系统将其结构化为：

- 公司
- 岗位
- 轮次
- 结果
- 复盘摘要
- 面试官问题
- 考察意图
- 回答卡点
- 下次策略
- 可复用证据
- 标签

这些内容会被写入数据库，并生成向量用于后续检索。

### 4.5 历史记忆管理

历史页支持查看、展开和删除 Job Memory。用户可以看到每次复盘中的问题、卡点、策略和证据。

设计重点：让 RAG 的记忆库可见、可管理、可信，而不是一个黑盒。

## 5. 技术实现

### 前端

- 微信小程序：承载手机端核心流程。
- H5 页面：用于浏览器访问和调试。

### 后端

- Node.js 原生 HTTP 服务。
- PM2 常驻运行。
- 腾讯云轻量应用服务器部署。

### AI 与数据

- DeepSeek：用于 JD 分析、问答生成、复盘结构化。
- Embedding API：用于生成 Job Memory 和查询向量。
- Supabase Postgres + pgvector：用于存储和向量召回。

### 访问控制

- 使用 `APP_ACCESS_TOKEN` 做轻量访问保护。
- 使用 `APP_INVITE_USERS` 将不同邀请码映射到不同 `user_id`。
- 后端按 `user_id` 隔离复盘保存、历史列表、删除权限和 RAG 召回。

## 6. RAG 设计

### 写入阶段

1. 用户粘贴面试复盘。
2. DeepSeek 将复盘结构化为 Job Memory。
3. 后端抽取摘要、问题、卡点、策略和证据。
4. Embedding API 生成向量。
5. Supabase 存储结构化数据和向量。

### 检索阶段

1. 用户输入面试问题或请求准备本轮面试。
2. 后端根据问题、JD 和轮次构造 query。
3. 生成 query embedding。
4. 使用 pgvector 召回相似 Job Memory。

### 生成阶段

1. 后端将召回的记忆拼入 prompt。
2. DeepSeek 结合当前问题和历史记忆生成回答。
3. 前端展示回答和“本次参考记忆”。

## 7. 关键取舍

### 邀请码优先，而不是完整账号系统

当前项目目标是个人使用和小范围内测，完整账号体系成本较高。因此先用邀请码实现轻量访问控制和用户隔离。

### 先做 Job Memory，而不是自动投递

自动投递容易偏离核心价值，也涉及更多平台规则和隐私风险。本项目优先解决面试准备和经验复用。

### 强约束 AI 不编经历

Prompt 中明确要求：如果没有简历或历史记忆证据，不把能力写成用户已经具备，不虚构数据和实验经历。
