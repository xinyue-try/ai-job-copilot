# AI Job Copilot 需求文档

## 1. 项目背景

求职过程中，候选人通常会遇到三个连续问题：

- 投递前：不知道一个 JD 是否值得投，也很难快速判断岗位风险点。
- 面试前/面试中：遇到追问时容易临场卡住，回答缺少结构和个人经历支撑。
- 面试后：复盘内容容易散落在聊天记录、文档或脑子里，下一次面试很难复用。

AI Job Copilot 的目标不是做一个泛泛的求职平台，而是面向个人真实求职流程，构建一个从 JD 分析、面试准备、即时问答到复盘沉淀的闭环工具。

## 2. 产品定位

AI Job Copilot 是一个个人求职面试 Copilot。

它帮助候选人完成：

- 投递前判断岗位匹配度和风险。
- 面试前基于 JD 和历史复盘生成准备方案。
- 面试中/面试前快速生成回答思路。
- 面试后把复盘结构化沉淀为 Job Memory。
- 后续通过 RAG 召回历史经验，辅助下一次面试。

一句话定位：

> 一个面向个人求职流程的 AI Copilot，把“投递前 -> 面试中 -> 面试后复盘 -> 下次准备”做成可持续迭代的闭环。

## 3. 目标用户

### 核心用户

- 正在找产品、AI 产品、数据产品、策略产品、用研等岗位的候选人。
- 需要频繁分析 JD、准备面试和复盘面试的人。
- 希望把自己的面试经验沉淀下来，而不是每次重新准备的人。

### 当前使用者

项目当前优先服务个人求职场景，不做开放平台。

## 4. 核心场景

### 场景一：投递前分析 JD

用户粘贴 JD 或上传岗位截图，系统结合用户简历分析：

- 岗位摘要
- 匹配分
- 主要匹配点
- 主要风险点
- 高风险面试雷达
- 缺失能力
- 简历优化建议
- 打招呼语

目标是帮助用户判断：

- 这个岗位值不值得投。
- 面试官最可能追问什么。
- 当前简历和经历哪里可能被质疑。

### 场景二：基于历史记忆准备面试

在 JD 分析结果页，用户可以点击“用历史记忆准备本轮面试”。

系统会基于当前 JD、岗位信息和历史 Job Memory 生成：

- 本轮重点
- 历史风险
- 可能问题
- 回答策略
- STAR 回答提醒
- 练习计划
- 一分钟 Pitch

目标是让历史复盘真正参与下一次面试准备，而不是只存在数据库里。

### 场景三：问一问 AI

用户输入面试问题，系统结合：

- 当前 JD
- 用户选择的轮次
- 回答风格
- 回答时长
- 历史 Job Memory

生成：

- 30 秒短答
- 完整答法
- 可引用证据
- 不要这样说
- 可能追问
- 本次参考记忆

目标是帮助用户把回答从“泛泛 AI 建议”变成“结合自己经历的表达”。

### 场景四：面试后复盘沉淀

用户粘贴面试复盘或原始 ASR/实录文本，系统结构化为 Job Memory：

- 公司
- 岗位
- 轮次
- 结果
- 复盘摘要
- 面试官问题
- 考察意图
- 回答卡点
- 下次策略
- 可复用证据
- 标签

系统将结构化结果写入 Supabase，并生成向量用于后续检索。

### 场景五：历史记忆管理

历史页展示已经沉淀的 Job Memory。

用户可以：

- 查看每条记忆摘要
- 展开查看问题、卡点、策略和证据
- 删除重复或无效记忆

目标是让 RAG 的“记忆库”可见、可管理、可信。

## 5. 功能模块

### 5.1 JD 分析模块

输入：

- JD 文本
- 岗位截图 OCR 结果
- 用户简历文本

输出：

- 岗位摘要
- 匹配分
- 风险点
- 面试雷达
- 打招呼语

### 5.2 问一问 AI 模块

输入：

- 面试问题
- 面试轮次
- 回答时长
- 回答风格
- 当前 JD
- 历史记忆召回结果

输出：

- 短答
- 完整答法
- 证据
- 避坑表达
- 可能追问
- 参考记忆

### 5.3 Job Memory 模块

输入：

- 面试复盘文本
- 公司
- 岗位
- 轮次
- 结果

处理：

- LLM 结构化
- Embedding 向量化
- Supabase pgvector 存储

输出：

- 可检索的 Memory Card

### 5.4 RAG 检索模块

触发场景：

- 问一问 AI
- 基于历史记忆准备本轮面试

流程：

1. 根据当前问题/JD/轮次构造检索 query。
2. 生成 query embedding。
3. 从 Supabase pgvector 召回相似 Job Memory。
4. 将召回结果拼入 prompt。
5. LLM 生成结合历史经验的回答。

### 5.5 邀请码访问控制与用户隔离

当前版本面向个人使用，后端支持 `APP_ACCESS_TOKEN`。

小程序首次使用需要输入邀请码，后续请求自动带上：

```text
x-app-token
```

用于避免陌生人消耗 API Key 或访问个人数据。

如果需要给多人试用，可以通过 `APP_INVITE_USERS` 将不同邀请码映射到不同 `user_id`：

```text
AJC-YOU-9Q7X -> you
AJC-FRIEND-2K8M -> friend
AJC-DEMO-7P3Q -> demo
```

后端会按 `user_id` 隔离 Job Memory：

- 历史列表隔离
- 复盘保存隔离
- 删除权限隔离
- RAG 召回隔离

这样别人试用时不会污染个人面试记忆。

## 6. 非功能需求

### 隐私

- API Key 只能存在服务端环境变量。
- Supabase service role key 不能进入小程序端。
- 邀请码不等同于完整账号系统，只用于轻量访问保护。
- 不建议公开给陌生用户使用。

### 稳定性

- 后端部署在腾讯云轻量服务器。
- 使用 PM2 常驻运行。
- `/api/health` 用于检查 DeepSeek、Supabase、Embedding 等依赖状态。

### 可维护性

- 后端 Node.js 单服务。
- 小程序端通过统一 `utils/api.js` 管理 API 地址和 token。
- 数据库结构通过 `supabase-memory.sql` 维护。

## 7. 当前版本边界

当前版本不做：

- 自动投递
- 多用户账号体系
- 支付体系
- 简历自动改写并投递
- 对外开放平台

当前版本重点是：

- 个人求职闭环
- RAG 记忆复用
- 小程序可用形态
- 云端内测部署
